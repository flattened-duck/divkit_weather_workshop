## Слайд 5 — AppHost: запрос как граф

- [притормози] Самый важный слайд раздела — потратить ~3 минуты на диаграмму.
- AppHost — оркестратор: запрос разбивается на граф, каждая вершина — отдельный сервис.
- bromapi никогда не ходит за данными сам — Morda-Go приносит готовые Protobuf-блоки.
- [нить: данные не наши] «Граф конфигурируется снаружи — bromapi не знает, кто ещё в нём живёт.»
- [вопрос залу] Риторически: «А что, если Morda-Go упала? Пустой ответ?» — не отвечать, вернёмся на слайде 19.

## Слайд 6 — Изнутри bromapi: Servants и Renderers

- [аналогия] Servant = Activity/Fragment (точка входа, один на экран); Renderer = Composable/ViewHolder (переиспользуемый кусок).
- AppHostService → Servant → пачка Renderers → JSON — линейная труба.
- Renderer получает данные, не запрашивает их — это принципиально.
- Koin: request-scope изолирован от singleton-scope, нет утечек состояния между запросами.

---

## Слайд 7 — Откуда данные: MADM и граф

- MADM — не база данных в классическом смысле: это «настройки продукта» (фиды, div-карточки, AB-флаги).
- bromapi получает готовые Protobuf-блоки: user_profile_block, morda_header_block...
- `DivDataBlock` — готовая DivKit-карточка из MADM, бэкенд просто пробрасывает без разбора.
- [нить: данные не наши] «Бэкенд конфигурируется снаружи — это намеренное архитектурное решение.»

## Слайд 8 — Модули bromapi: карта проекта

- Быстро, 1–1.5 минуты, не перечислять всё подряд.
- Главное правило: `app/` — только servants и интерфейсы; реальная логика — в модулях.
- Код в `app/` вне интерфейсов/servants — это техдолг, CI об этом знает.
- Ориентир: `api/` — вход/выход, `renderer-commons/` — шаренная логика, `palette/` — цвета.

## Слайд 9 — DivKit DSL: как пишем вёрстку на Kotlin

- DSL на Kotlin, но на выходе — JSON; клиент Kotlin не видит.
- `divan { }` — top-level builder, внутри — `container()`, `text()`, `image()`, `gallery()`.
- [аналогия] Templates ≈ RecyclerView + ViewHolder: структуру объявляешь раз, payload — только данные.
- Expressions: `variable equalTo "value"` → другое поведение/цвет/видимость без нового релиза.
- [переход] «Поняли, как выглядит изнутри. Теперь — один запрос = один экран?»

## Слайд 10 — [РАЗДЕЛ] Несколько экранов

- [вопрос залу] «Когда открываете Браузер — сколько запросов к бэкенду? Угадайте.»
- Ответ: один (в идеале). Один AppHost-запрос — несколько документов.
- «Каждый документ — независимый DOM с переменными и своим состоянием.»

## Слайд 11 — Multiple screens: RequestType → Document

- RequestType разделяет поверхности: MORDA, SEARCHAPP_PROFILE, BOTTOM_SHEET, YAZEKA — разные Servants.
- Диаграмма: один запрос → четыре Servant → четыре независимых документа.
- FlexNtpDocumentLoader управляет жизненным циклом каждого документа независимо.
- Preloader: документы начинают грузиться до того, как пользователь их открыл — нет задержки на показ.

## Слайд 12 — Предзагрузка и skeleton

- [нить: fallback chain] Skeleton — первый уровень цепочки: до того, как документ готов.
- Skeleton — тоже DivKit JSON, зашитый в APK; никакого нативного View для лоадера.
- SkeletonContentController: показывается до первого успешного ответа, потом сменяется реальным документом.
- Cache-first ещё не раскрыли — скоро, в блоке состояния.
- [переход] «Документ загрузился, пользователь тапнул. Нужны ещё данные — дозапросы.»

## Слайд 14 — NetworkRequestAction: дозапрос из DivKit

- Действие пользователя (тап, скролл) → DivKit-action → NetworkRequestAction.
- NetworkRequestActionHandler на клиенте делает HTTP-запрос — клиент не знает, что за URL.
- ReferenceReplaceSectionAction: точечная замена секции без перерисовки всего экрана.
- [нить: бэкенд объявляет] «URL и поведение после ответа — всё описано бэкендом в JSON.»

---

## Слайд 15 — Паттерн «дозапрос с состоянием»

- [притормози] Четыре шага — проговорить каждый, это самый важный паттерн раздела.
- Шаг 1: переменная `is_loading = false` в вёрстке от бэкенда.
- Шаги 2–3: тап → SetVariableAction + NetworkRequestAction → спиннер через expression.
- Шаг 4: onSuccessAction обновляет данные и сбрасывает `is_loading`.
- Весь UX загрузки описан декларативно в JSON — ни строчки нативного кода на клиенте.
- [вопрос залу] «А если запрос упал с ошибкой — кто обрабатывает?» — onErrorAction тоже декларативный, или дефолтный error state клиента.
- [переход] «Временное состояние разобрали. Есть состояние, которое живёт дольше.»

## Слайд 16 — [РАЗДЕЛ] Состояние и хранение

- [нить: бэкенд объявляет] «Кто хранит состояние? Клиент. Кто его объявляет? Бэкенд.»
- [аналогия] Как ViewModel хранит StateFlow, а API-ответ диктует начальное значение — только здесь это DivKit-переменные.

## Слайд 17 — DivKit Variables: объявление и использование

- [притормози] Ключевой слайд всей лекции — потратить достаточно времени.
- [аналогия] Variables ≈ LiveData/StateFlow: бэкенд объявляет `StringVariable("theme", "light")` — как `MutableStateFlow("light")`.
- Клиент обновляет через DivVariableController, DivKit пересчитывает expressions автоматически.
- CommonVars — это shared contract: и бэкенд, и клиент знают эти имена.
- [нить: переменная как контракт] «Переименовать `theme` в `color_theme` — breaking change. Клиент сломается.»

---

## Слайд 18 — Кэширование на клиенте

- [аналогия] Fallback chain ≈ Room + network + hardcoded defaults — те же слои, другие технологии.
- Cache-first стратегия: показать кэш немедленно, обновить в фоне.
- FlexNtpNetworkCaching → disk cache; LazyNetworkCaching обновляет только если экран открыт.
- Пользователь никогда не ждёт с нуля — это требование, не опция.

---

## Слайд 19 — Зашитая вёрстка (Hardcoded Fallback)

- [нить: fallback chain] Закрыть вопрос со слайда 5: «Что если данные не пришли?»
- `searchapp_flex_ntp.json`, `mobileyandex_div_profile.json` — лежат в `res/raw/` APK.
- Цепочка: network → disk cache → raw JSON в APK → skeleton → error state.
- [вопрос залу] «Зашитый JSON устаревший — не страшно?» Ответ: актуальность vs. доступность. Лучше устаревший, чем пустой экран.
- Hardcoded JSON — страховка, не фича. Обновляется только с релизом APK.

## Слайд 20 — Офлайн: OfflineNtpFeature

- Offline — не «ошибка сети», а отдельный content-type — планировать заранее.
- disk cache → fallback JSON → offline-режим (игры, сохранённые страницы).
- OfflineNtpFeature: `preload_games = true` — игры грузятся заранее, пока есть сеть.
- [переход] «Состояние разобрали. Особый вид состояния — тема интерфейса.»

## Слайд 22 — Как работает темизация в DivKit

- Диаграмма — главное: System Dark Mode → DarkThemeStateController → DivPaletteUpdater → variableController → DivKit пересчитывает.
- Смена темы = одна операция: обновить все Color-переменные сразу.
- Palette: именованные цвета — `icon/default/secondary`, `bg/primary` — имена те же на бэкенде и на клиенте.
- DivPaletteUpdater следит за темой; рендеринг пересчитывается автоматически.

## Слайд 25 — Как работает локализация в bromapi

- Четыре шага — отдельный конвейер, независимый от bromapi и от APK.
- Tanker → lang-auto.json → релиз Morda Localization → REX доставляет runtime (~несколько минут).

- [вопрос залу] «Переводчик ошибся в тексте — как быстро исправить без APK-релиза?» Несколько минут через REX.
- [нить: без синхронного релиза] Переводы живут независимо от релиза сервиса — это архитектурная особенность.
- [переход] «Runtime-обновления умеем. Теперь задача потяжелее: клиент старый, бэкенд уже новый.»

## Слайд 27 — Три уровня версионирования

- Три уровня независимы: Протокол / Layout-DSL / Бизнес-логика — можно менять по отдельности.
- [аналогия] Версионирование ≈ API Level checks: `if (Build.VERSION.SDK_INT >= 33)` — то же самое, только спрашивает бэкенд, а не клиент у ОС.
- Protobuf: поле может стать obsolete, но не может быть сломано без предупреждения.
- Бизнес-логика: minVersion-проверка в Servant + AB-флаги закрывают «только для >= N».

## Слайд 28 — Версионирование layout: что клиент не поймёт

- [притормози] Практический чеклист — говорить медленно.

- Опасно: удалить обязательное поле, изменить семантику action, изменить имя переменной.
- «Всегда думайте: что увидит клиент двухлетней давности?»

## Слайд 33 — Паттерны бэкенда: Servant + Renderer

- Servant per Surface: один класс — один тип запроса; нет god-class, обрабатывающего всё.
- Renderer Composition: экран = сумма маленьких Renderer-ов; каждый тестируется через `RendererTest<IMyRenderer>`.
- Template DSL: повторяющиеся блоки → template + reference + defer; payload уменьшается в разы.
- No Business Logic in Renderer — правило, которое легко нарушить: логика в Adapter, не в Renderer.

---

## Слайд 34 — Паттерны бэкенда: данные и зависимости

- Koin Request Scope: зависимости живут ровно один запрос — нет утечек состояния.
- Interface + @RequireTests: каждый renderer обязан иметь интерфейс и интеграционный тест; CI проверяет автоматически.
- [нить: переменная как контракт] «Variables as Contract: изменить имя переменной = breaking change. Это публичный API.»
- Структура кода — про поддерживаемость через год, не про красоту сегодня.

## Слайд 35 — Паттерны клиента: Document lifecycle

- Cache-First + Background Refresh: показать кэш немедленно, обновить в фоне — повторение блока 4.
- Fallback Chain: network → disk cache → hardcoded JSON → skeleton → error — цепочка завершена.
- Variables as Single Source of Truth: состояние UI только через DivVariableController; нет параллельных state-holders.
- [нить: бэкенд объявляет] «Клиент тупой — это хорошо. Вся логика на сервере, клиент не должен знать о бизнес-логике.»

## Слайд 36 — Паттерны клиента: тема и offline

- Theme as Variables: тема не перерисовывает View — обновляет Color-переменные, DivKit пересчитывает сам.
- Offline как content-type: OfflineNtpFeature — не «ошибка сети», а полноценный экран с контентом.
- Skeleton as DivKit: loading-анимация — тоже DivKit JSON; нет нативного View для лоадера.
- «Пользователь никогда не видит пустой экран» — финальный тезис раздела.
- [переход] «Всё. Паттерны назвали. Осталось — три вещи на вынос.»

## Слайд 37 — Итог: что взять с собой

- «Запомните три вещи» — не «подведём итоги».
- Контракт: бэкенд объявляет, клиент исполняет. Переменная — это публичный API, не просто конфиг.
- Fallback chain: всегда думайте о следующем уровне. Нет сети → кэш → JSON → skeleton.
- Независимые релизы: переводы, флаги, AB — всё умеет жить без синхронного релиза APK и сервиса.
- Большой проект отличается от tutorial именно этими темами: версионирование, офлайн, AB — не «потом добавим».
