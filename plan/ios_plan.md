# iOS client — implementation plan

Goal: a native **iOS** client with feature parity to the existing Android client (`app/`), rendering the SAME backend envelope from the SAME Ktor server, on **open DivKit iOS**. This plan is for an implementing agent with no prior context. It deliberately does NOT restate what already lives in the repo or the DivKit sources — it points at them.

## Where the spec already lives (READ, don't reinvent)
- **Client↔server contract & name map** — `plan/contract.md` (envelope shape, the `/document?lang=&lat=&lon=&name=`, `/city-search`, `/zero` endpoints, the DivPatch shape, `bg_key` grammar + raw-URL pattern, and the cross-client "name map": `custom_type` `sun_phase` with `custom_props {sunrise,sunset}` (client computes "now"); extension id `scroll_state`; global vars `theme`/`theme_mode`/`compact`/`header_state`/`status_inset`/`nav_inset`; container id `city_search_results`; var `city_query`; action URLs `weather-app://navigate|back|set_lang|set_theme|set_compact|set_city|city_search`). These are the CONTRACT the iOS client must honor verbatim — same as Android.
- **The behavioral reference is the Android client** `app/src/main/java/com/example/weatherdivkit/` — read `MainActivity.kt`, `WeatherDivActionHandler.kt`, `divkit/DocumentLoader.kt`, `divkit/ScrollStateExtensionHandler.kt`, `divkit/SunPhaseCustomViewAdapter.kt`. Port BEHAVIOR, not code. The subtle offline/UX rules are in the memory `app-requirements` and encoded there.
- **Backend is unchanged** — do NOT modify `backend/`. Run it (`cd backend && ./gradlew run`, `:8080`). The zero skeleton + `scripts/update-bundled-layout.sh` already exist; the iOS bundled fallback asset is the same `/zero` output (see Offline below).
- **DivKit iOS source (verify every API here, not by guessing):** `/Users/the-leo/divkit_source/divkit/client/ios` @ branch **`R-32.57`** (== 32.57.0, the pinned version). Canonical "how to host DivKit" examples: `client/ios/Samples/` and `client/ios/DivKitPlayground/` — mirror their `DivKitComponents` setup. Non-obvious DivKit facts (shared with Android) are in the memory `divkit-android-gotchas`; re-verify each on the iOS side.

## Toolchain / setup
- Swift Package Manager. Add DivKit via its `Package.swift` (products: `DivKit`, `DivKitExtensions`, and `LayoutKit`/`Serialization` come transitively) pinned to the **`32.57.0`** tag (match the backend/Android version). Prefer depending on the published package; if the tag isn't on the SPM registry, point at the local checkout or the GitHub `divkit/divkit` at that tag.
- Host in **UIKit** following the Samples (DivKit renders a `DivView`/block into a UIView via LayoutKit; SwiftUI wrapping is optional). Min iOS: match the Samples' deployment target.
- **Simulator networking differs from Android:** the iOS Simulator reaches the host backend at `http://localhost:8080` (or `127.0.0.1`), NOT `10.0.2.2`. Make the base URL configurable. ATS: `localhost`/HTTP is allowed by default in the Simulator; `raw.githubusercontent.com` images are HTTPS (fine). There is no emulator Wi-Fi proxy on iOS, so the Android `Proxy.NO_PROXY` workaround is NOT needed.

## The core: Android mechanism → DivKit-iOS equivalent
For each, the DivKit-iOS anchor file is named; open it + the Samples to get the exact current signature.

1. **Render the envelope, 3 screens, navigate without refetch.** Parse `templates` once and build a `DivData`/card per screen; keep them in memory; navigation swaps the active card in the hosting `DivView`. Entry point: `DivKit/DivKitComponents.swift` (the analog of Android's `DivConfiguration`) + how the Samples build a `DivView` from `RawDivData`. Refetch only on `set_lang`/`set_city` (see Offline for the graceful rules).
2. **`weather-app://` actions.** Implement a custom URL handler and pass it to `DivKitComponents`. Anchors: `DivKit/Actions/DivUrlHandler.swift`, `DivActionURLHandler.swift`, and `DivKitComponents`' `urlHandler` parameter. Handle the same hosts as Android's `WeatherDivActionHandler`: navigate/back/set_lang/set_theme/set_compact/set_city/city_search.
3. **Global reactive variables** (`theme`, `theme_mode`, `compact`, `header_state`, `status_inset`, `nav_inset`). Use DivKit's variable storage (the `DivVariablesStorage` / global variables passed into `DivKitComponents`; see `Actions/SetVariableActionHandler.swift` and the Samples' variable setup). Declare them once globally and mutate them from native code (theme/compact from settings actions; `header_state` from the scroll extension). `theme`/`compact` are reactive → no refetch; `set_lang`/`set_city` refetch.
4. **`scroll_state` extension → header collapse.** Implement a `DivExtensionHandler` (protocol `DivKit/DivExtensionHandler.swift`; working examples in `DivKitExtensions/ExtensionHandlers` and `SizeProvider`). Attach to the vertical gallery carrying `extensions:[{id:"scroll_state"}]`; observe its scroll offset and write the global **`header_state`** String var ("full"/"collapsed"). Port the Android hysteresis: collapse past ~24pt, expand only when back at the very top (iOS scroll-view `contentOffset.y <= 0`), re-check on scroll-end. The backend header uses `state_id_variable="header_state"` — verify iOS `state` reacts to the variable the same way.
5. **Shimmer — USE THE BUILT-IN.** DivKit iOS ships a shimmer extension: `DivKitExtensions/Shimmers`. Register that handler in `DivKitComponents` (no custom code, unlike Android which added a dependency). The zero skeleton's shimmer blocks are `image` divs at the DivKit `empty://` sentinel — confirm the iOS shimmer keeps animating on an unloaded/`empty://` image and produces no error (verify against `Shimmers` + `DivKit/Images`).
6. **`sun_phase` DivCustom.** Register a custom block factory (search `DivKit/Extensions/DivCustom*` + `DivCustomBlockFactory` and the Samples' custom-block example) that draws the sunrise→sunset arc with Core Graphics from `custom_props {sunrise,sunset}` ("HH:mm"; compute "now" from the device clock). **Gotcha (same as Android):** an arc drawn into a 2r×r rect is an ELLIPSE — place the marker on that ellipse (`my = (cy - r/2) - (r/2)·sin θ`), not a circle.
7. **Images (backgrounds, popup, base64 preview).** DivKit iOS loads images via `DivKit/Images/DivImageHolderFactory.swift` + `CachedRemoteImageHolder.swift` (remote caching is built-in). Wire the default image holder; the theme-driven day/night background URL and the popup `scale=fit` + base64 `preview` are already in the backend layout — nothing client-side except providing the image loader.
8. **DivPatch (city search).** `city_search` action → fetch `GET /city-search?q=&lang=` → apply the returned patch to the live card. Anchors: `DivKit/Patches/` (`DivPatchProvider`, `DivPatchDownloader`) and how the Samples apply a patch to a `DivView`/`DivData`. The patch targets container id `city_search_results` (note the "patch replaces" gotcha — the backend already wraps the rows in a same-id container). Read `q` from the resolved `city_query`; URL-encode it for the backend request.
9. **Insets.** iOS has `DivKit/DivSafeAreaManager.swift` — check whether enabling it makes the `status_inset`/`nav_inset` variables unnecessary on iOS (the backend paddings read those vars via `edgeInsets(...).evaluate`). Either feed `view.safeAreaInsets` (pt) into the `status_inset`/`nav_inset` global vars to reuse the existing layout as-is, OR rely on DivSafeAreaManager — pick one and keep the header/FAB/settings paddings correct. Verify on a notched device.
10. **Status bar theming.** Follow the effective theme: set `preferredStatusBarStyle` (`.lightContent` on dark theme, `.darkContent` on light) and refresh via `setNeedsStatusBarAppearanceUpdate()` on theme change (analog of Android's `isAppearanceLightStatusBars`).
11. **Pull-to-refresh.** Use `UIRefreshControl` on the main screen's scroll container (or the DivKit gallery's underlying scroll view) — enabled on main only; on refresh, refetch `/document` for current lang+city and swap on success (spinner always ends). DivKit has no PTR (same as Android). Finding the right scroll view: DivKit galleries are UICollectionView-backed; locate the main scroller (the div id `main_scroll` — check how iOS DivKit exposes a div id on its view, analog of Android's view-tag).

## Offline: cache + zero skeleton + graceful language switch (port the Android rules exactly)
The intended behavior is in memory `app-requirements` and `app/.../DocumentLoader.kt` + `MainActivity.kt`. Replicate on iOS:
- **Cache** the last successful network body to `Caches`/`Documents` keyed by lang (e.g. `doc_cache_<lang>.json`), written on parse success.
- **Cold-start priority:** network → cache(lang) → bundled zero skeleton, with a two-phase start (show cache/skeleton instantly, fetch network in the background, swap on success).
- **Bundled zero skeleton:** ship the `/zero?lang=ru` output as a bundled resource (generate it with the existing `scripts/update-bundled-layout.sh` pattern, or a tiny iOS-side script that curls `/zero` into the app bundle). It carries dashes + `empty://` shimmer images — no real data.
- **Graceful refresh (lang/city/PTR):** network-ONLY; on failure KEEP the current layout (never fall through to bundle/cache — that fall-through is the layout-wipe bug). `set_lang` may try the lang cache before giving up.
- **No connectivity auto-refetch** — do NOT add an `NWPathMonitor` refetch. Refresh is manual.

## Suggested stages (each: implement → run in Simulator → verify parity)
0. SPM setup + host a `DivView` from a hardcoded `/document` fetch (one screen renders). Confirm base-URL = `localhost:8080`.
1. Envelope → 3 screens + navigation + the `weather-app://` URL handler + global variables (theme/compact reactive, lang/city refetch).
2. Native extensions: `scroll_state` header collapse + register built-in Shimmer; `sun_phase` custom block (ellipse marker).
3. Images + theme day/night background + status-bar theming + insets (safe area).
4. City search (input → `/city-search` → DivPatch apply) + pull-to-refresh.
5. Offline: cache + two-phase start + zero skeleton bundle + graceful lang/city switch.
6. Parity pass: light/dark, compact, ru/en, offline cold-start (cache vs skeleton), city switch, PTR — matching the Android screenshots.
